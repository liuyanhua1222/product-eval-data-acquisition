# SKILL.md — 院外新产品筛选及评估执行技能

## 定位

本 Skill 是 **3.2 版院外新产品筛选及评估框架**的完整执行手册。

**输入：** 产品名称（和/或批准文号）
**输出：** 三阶段完整评估报告（GO/NO GO 结论 + 销售预测 + 最终放行）

> ⚠️ **首次使用前必读**：请先阅读同目录下的 **DEPENDENCIES.md**，确认所有平台账号、软件环境、数据源权限已就绪，否则部分数据无法获取。

---

## 技术栈

| 组件 | 用途 |
|------|------|
| `browser` tool（OpenClaw内置） | 主要数据获取方式，适合所有动态页面 |
| `session-manager.js`（本Skill） | 登录+Cookie持久化，维持会话状态 |
| `playwright-stealth.js`（playwright-scraper-skill） | 反爬虫站点增强爬取 |
| `web_fetch` tool（OpenClaw内置） | 静态页面快速获取（注意：国内医药类站点绝大多数被封锁） |

## 平台验证标准流程（每次评估前必做）

> ⚠️ **必须先登录，再验证数据获取，最后才执行评估事项。**
> 登录前的验证结论无效。

### 支持的登录方式

| 方式 | 命令 | 适用场景 |
|------|------|---------|
| **手动扫码/任意登录** | `login-manual <平台>` | ⭐推荐！支持所有平台所有登录方式（扫码/账号/短信），自动检测成功 |
| **Chrome标签页接管** | browser tool (profile="chrome") | Chrome已安装OpenClaw Browser Relay插件，标签页已attached |
| **自动化填表** | `login <平台> <用户名> <密码>` | 有固定账号，密码可在命令中传递 |
| **Cookie注入** | `login-cookie <平台> <cookie>` | 从浏览器DevTools复制Cookie字符串直接注入 |

> 💡 **推荐流程**：先试 `login-manual`（最简单）；若平台要求扫码或短信，选它最灵活。

### 验证流程

```
Step 1. 选择登录方式
├── 扫码/短信登录 → login-manual（自动检测成功）
├── Chrome已登录 → browser tool (profile="chrome")
├── 有账号密码 → login（自动填表）
└── 已有Cookie → login-cookie
↓
Step 2. 执行登录
├── login-manual：自动打开浏览器+检测+保存Cookie
├── browser tool：attached Chrome标签页后直接用
├── 登录成功后运行 status 确认会话有效
↓
Step 3. 登录后数据验证（用真实药品"阿司匹林"检索）
├── 确认能否获取规则所需的完整字段
├── 能获取 → 标注✅+可覆盖的规则编号
├── 字段缺失/权限不足 → 标注❌+原因
↓
Step 4. 更新"平台会话状态"表
└── 汇总登录方式、覆盖规则、缺口
```

## 凭证与会话管理

### 凭证文件

凭证保存在 `credentials.md`（**不要分享此文件**）：

```bash
cp credentials-template.md credentials.md
# 然后编辑 credentials.md
```

凭证格式：
```
【开思CHIS】
username: your_phone_or_email
password: your_password
last_updated: 2026-04-22
```

### 会话持久化机制

```
每次执行评估
    ↓
读取 credentials.md 获取账号
    ↓
检查 ~/.agent-browser/sessions/<平台>-cookies.json 是否存在且有效
    ├── 已有效 → 直接使用Cookie恢复会话，跳过登录
    └── 无效/不存在 → 执行自动化登录 → 保存Cookie
    ↓
执行数据获取（browser tool 或 playwright脚本）
    ↓
Cookie在会话结束后保留，下次无需重新登录
```

**会话状态查看：**
```bash
node scripts/session-manager.js status yaozh   # 查看药智网会话
node scripts/session-manager.js status kaisi  # 查看开思CHIS会话
node scripts/session-manager.js status douyin  # 查看抖音会话
```

**清除会话（需要重新登录）：**
```bash
node scripts/session-manager.js clear yaozh
```

## 执行输出规范

> 所有事项执行完成后，按以下结构输出结果。结构统一，便于后续汇总和复核。

### 标准输出字段

| 字段 | 说明 | 填写示例 |
|------|------|---------|
| **适用性** | 本事项是否适用于当前产品/场景 | 适用 / 不适用 / 需补充输入 |
| **判断结果** | 在适用前提下，按判断标准得出的结论 | 通过 / 不通过 / 需补充数据 |
| **关键数值** | 阈值、比例、排名、金额、人数、趋势等可复核数字 | `价差率=32%` / `Top1市占率=67%` / `同API批文数=3家` |
| **证据** | 来源名称 + 查询关键词 + 查询日期 + 链接或截图说明 | `药智网批文查询，"门冬氨酸钙"，2026-04-22，db.yaozh.com/pijian` |
| **判断说明** | 一句话说明通过、不通过或需补充的原因 | `同API仅1家企业持有有效批文，独家窗口不低于3年，通过` |
| **需补充来源** | 当前公开/平台路径无法完全闭环，需人工或内部数据补充 | `转化率需京东商家后台内部数据` |

## 逐条规则数据验证方法论

> ⚠️ **平台能打开 ≠ 数据能采集。必须对每条规则逐条验证实际数据获取能力。**

### 验证原则

| 原则 | 说明 |
|------|------|
| **选一个具体产品** | 固定用"阿司匹林"作为测试用例，便于交叉验证所有规则 |
| **明确字段清单** | 每条规则需哪些字段，列清楚 |
| **实际抓取** | 在对应平台实际执行搜索/查询，抓到数据才算 |
